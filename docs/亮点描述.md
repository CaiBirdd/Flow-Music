# 亮点描述

## 亮点一：LRC 歌词解析与播放组件

核心功能：封装 LRC 歌词解析与播放组件，支持多时间标签及双语歌词匹配；采用二分查找定位当前播放歌词行，基于 requestAnimationFrame 实现与刷新率同步的时间驱动；配合 GSAP 实现平滑滚动并处理用户手动滚动歌词冲突。

### 1. LRC 歌词解析（parser.ts）

LRC歌词解析有个特殊情况：一行LRC歌词可能存在多个时间标签，比如 `[01:10.00][02:10.00]都是同一句歌词`。如果用普通的正则去 exec，匹配到一个时间标签它就停了。 所以我给时间正则加上 g 全局标识，配合 while 循环把当前LCR歌词行所有的时间标签全部取出，同时用正则的 lastIndex 记录一下最后一个标签的位置，方便把时间标签后面的歌词文本截取出来。提取完后，遍历所有时间标签，为每个标签匹配一个独立的歌词文本对象。最后用sort对整个数组按时间升序排序（因为遍历后时间的顺序可能是乱的），再通过相邻两行的时间差算出每行的持续时长 `duration`，方便后面做滚动的高亮匹配。

对于双语歌词匹配，因为翻译歌词其实也是 LRC 格式，所以解析复用了同一parseLRC函数。然后将解析好的翻译歌词数组建成一个Map哈希表，用时间作为key，翻译文本作为 value。 遍历原歌词去 Map 里用Map.has查，时间相同就能直接匹配到，然后就将翻译文本赋值给原歌词对象的translation属性;有时存在原歌词时间和翻译歌词时间存在误差的情况，这里做了个容差匹配，这种情况下会遍历整个Map表，找时间差 ≤ 0.5 秒歌词进行组合。
一开始最简单的想法其实是双层 for 循环找，但时间复杂度为 O(n²)。改为哈希表后，查找就变成了 O(1)，整个匹配过程降维成了 O(n) 级别的单层遍历，算是一个小的性能优化点吧。

### 2. 二分查找定位当前行（player.ts - `findCurrentLine`）

播放时需要根据当前播放时间快速定位到对应的歌词行。歌词数组已经排好序了，所以我用了二分查找。但和普通二分查找不同的是，这里不是精确匹配某个值，而是区间匹配——判断当前时间是否落在 `[当前行时间, 下一行时间)` 这个左闭右开区间里。原因很简单：每句歌词都会持续一段时间，不是一个瞬时点。时间复杂度 O(log n)，比线性遍历高效很多。

### 3. requestAnimationFrame 时间驱动（player.ts - `timeLoop`）

requestAnimationFrame是浏览器用于实现高性能动画的 API，它的优势在于执行频率能和浏览器的刷新率保持同步。比如浏览器的刷新率为60hz时，它就每秒执行60次，每次更新一帧动画。
具体是，只要音乐在播放，就让requestAnimationFrame的每一帧去跑一个timeloop函数检测循环。在每一帧里，先拿到 audio 的当前播放时间，然后根据当前播放时间二分查找算出应该高亮的歌词行，最后是将这帧算出的行号index和上一帧的index作比较，只有变了才去更新歌词的高亮和平滑滚动。
因为歌词一句可能要唱好几秒，这段时间里如果rAF每帧都去更新 DOM，性能开销会很大。这样就规避了大量无意义的操作。

### 4. GSAP 平滑滚动 + 用户滚动冲突处理（player.ts - `scrollToLine`）

歌词的平滑滚动我用GSAP 的 `gsap.to()`来实现。
这里交互冲突问题是：如果用户正在手动滚动歌词浏览，自动滚动还在跑的话，页面就会"抢来抢去"。所以我加了一个 `isUserScrolling` 标记，并在歌词容器的最外层监听 `wheel`滚动事件，一旦用户滚动了就把标记设为 `true`，并启动一个 3 秒的防抖定时器，自动滚动逻辑看到这个标记就会return放弃接管。3 秒后用户没再滚动，标记恢复为 `false`，自动滚动继续接管。
另外，如果是用户点击歌词或者拖拽进度条，那就用自动滚动逻辑的`immediate` 模式直接跳转，不受这个标记影响。

### 5. 其他细节补充

- DOM 渲染用了 `DocumentFragment` 一次性插入，避免逐行 append 导致多次重绘
  const fragment = document.createDocumentFragment()
  this.container.appendChild(fragment)
- 事件委托：歌词的点击事件只在容器上绑一个，通过 `closest('.lyric-line')` 冒泡找到对应行

## 亮点二：动态视觉主题与多彩背景

核心功能：设计动态视觉主题，基于 ColorThief 从歌曲封面提取调色板，通过 RGB→HSL 转换过滤低质量颜色后同步至全局状态，驱动背景渐变与进度条样式联动；同时使用 Canvas 将封面切割生成流动背景，配合双缓冲渐变过渡实现平滑切歌换肤。

### 1. 开场总览

这个功能的核心目标是：每换一首歌，歌曲详情页的视觉风格都会跟着歌曲的封面变化——包括歌曲详情页背景渐变、叠了一个流动背景，还有底部进度条颜色。整体链路是：颜色提取 → 过滤筛选 → 全局广播 → UI 联动。

### 2. 颜色提取与过滤（utils/index.ts）与全局状态同步

首先通过 ColorThief 这个库从歌曲封面里提取一个调色板（palette），它会返回一组按权重排序的 RGB 颜色数组。但直接拿来用不太行，因为提取出来的颜色可能太黑、太白、太灰，做背景不太好看。所以我做了一轮 RGB → HSL 转换，在 HSL 的饱和度（S）和亮度（L）上做过滤：亮度低于 0.2 或高于 0.8、饱和度低于 0.2 或高于 0.8 的颜色都会被排除。

过滤后还需要从剩余的颜色里选出最终用于渐变的 2 个颜色。这里我用了一个贪心策略：首先按权重选第一个，也就是主色调，然后遍历剩下的，挑一个和主色差异最大的。比较差异时我只比较 S 和 L、忽略了色相 H，这样做是为了选出是有层次变化的同色系颜色，比如说选了一个主色是深蓝，第二个颜色就会选择浅蓝，亮蓝色这种同色系但有差异的。
选好的颜色通过 Pinia存入全局 store供其他组件使用。

### 3. Canvas 切割生成流动背景（useMusic.ts - `useRhythm`）

这里是把歌曲封面均分成 4 块，分别放在页面四角，然后让它们各自从随机角度开始做匀速旋转，最后加上一层 80px 的高斯模糊，这样视觉上就像一团团颜色在缓缓流动。
具体实现的话就是用了双重for循环遍历一个 2x2 的网格。每次循环，就拿原封面图去切割出一个小块画到 Canvas，再用 toDataURL 导出 base64 图片，拿来当作不同 div 的背景，最后用 CSS 配合一个随机生成的起始角度让它们转起来。
补充点：
性能优化上做了几件事：Canvas 对象池：在数组里预先 new 好了 4 个 Canvas，每次切歌时就直接取出来画，避免反复 `createElement` 带来的内存分配和 GC 压力
DOM 复用：只有第一首歌时我会去生成真实 DOM，后续切歌只要拿到现有的节点替换掉背景图和 CSS 角度参数。
整个绘图操作包在 `requestAnimationFrame` 里，确保在浏览器下一帧重绘前执行，避免阻塞主线程导致卡顿

### 4. 双缓冲渐变过渡（useMusic.ts - `gradualChange`）

因为切歌时直接替换背景会闪烁，所以我在页面上叠放了两个完全重合的 div（`#gradual0` 和 `#gradual1`），维护一个 `activeLayerIndex` 在这两层之间切换。每次切歌时，在透明的那层设置好新渐变，然后把它的 opacity 置为 1，旧层 opacity 置为 0。因为两个 div 都有 `transition: opacity 1s` 的 CSS 过渡，所以视觉上就是一个缓慢的渐变淡入淡出效果。

## 亮点三：全局音频控制器

核心功能：设计全局音频控制器统一管理 Audio 实例，供多个组件共享使用；通过方法劫持扩展原生 Audio API，实现播放/暂停时音量渐变的淡入淡出效果，提升听觉体验。

### 1. 全局音频控制器的设计

整个应用只有一个 `<audio>` 标签，放在底部的 `MusicPlayer`播放器组件中。问题是：有很多外部的地方都要用到它，比如进度条组件需要当前播放时间currentTime、音量组件还需它的volume。所以我最后选择把这个播放器实例挂载到window全局对象上，具体是在`MusicPlayer`播放器组件中通过 Vue 的 `defineExpose` 把 audio 元素和控制方法暴露出去，在最外层的 `App.vue`组件的 `onMounted` 里，通过 ref 拿到组件实例，赋值给 `window.$audio` 。这样做的好处是——不管是 Vue 组件还是 TS 文件——都可以通过 `window.$audio` 获取这个播放器实例。

### 2. 方法劫持实现淡入淡出（Monkey Patching）

首先我备份了原生 audio 的 play 和 pause方法，命名为originPlay，originPause。然后重写了它们。音量渐变的话是在函数中通过Promise 和setInterval实现： 在触发播放时，我先把音量设为 0，然后调用备份的originPlay让音频开始播放，通过setInterval每隔 50 毫秒的循环慢慢把音量加到目标值，这就实现了淡入。暂停则相反，先用setInterva将音量慢慢降到 0，等彻底静音后，再调用原生的originPause做真正的物理暂停，这就实现了淡出。

补充点：在静音后去触发物理暂停时，用 originPause.call(audio) 去调用原生方法，不能顺手写成 audio.pause()。因为pause方法已经被我们重写了，如果那样调用就会无限递归。

## 亮点四：组件休眠/唤醒机制

核心功能：针对歌曲详情页设计组件休眠/唤醒机制，页面不可见时降级为轻量渲染并销毁动画实例，避免隐藏组件的无效计算与渲染，后台 CPU 占用由约 15% 降至 <1%，显著提升桌面端运行稳定性。

应用的歌曲详情页是用 `transform: translateY(100%)+visibility:hidden` 来隐藏的，没有用 `v-if` 销毁组件。原因是这个页面结构比较重，涉及双缓冲渐变背景和流动背景、以及GSAP 动画等，如果每次都销毁重建，会很卡。

但这样也会有个问题，组件没销毁，`onUnmounted`就不会触发，即使用户没打开歌曲详情页，只在底部的播放组件切歌，一整套渲染和逻辑依然在跑。我的思路是用watch监听歌曲详情页的打开关闭状态， 当检测到详情页处于关闭状态时（休眠），组件只会执行轻量渲染，提取下主题色，不加载渐变背景和流动背景(return)，GSAP 动画也会调用kill进行销毁。当watch监听到用户打开详情页时（唤醒），就会针对当前歌曲补做完整渲染，GSAP动画会带上一个immediate=true的标识，跳过动画过程，展示最终状态，让用户感觉不到延迟。

补充：
raf在vue中就是放在生命周期里，onMounted启动、onUnmounted里销毁，cancelAnimationFrame
requestAnimationFrame 返回的是一个 ID
