# 亮点描述

## 亮点 1：LRC 歌词解析与播放组件

**核心功能**：封装 LRC 歌词解析与播放组件，支持多时间标签及双语歌词匹配；采用二分查找定位当前歌词行，基于 requestAnimationFrame 实现与刷新率同步的时间驱动；配合 GSAP 实现平滑滚动并处理用户手动滚动冲突。

### 一、LRC 歌词解析（parser.ts）

LRC歌词解析有个特殊情况：一行LRC歌词可能存在多个时间标签，比如 `[01:10.00][02:10.00]都是同一句歌词`。如果用普通的正则去 exec，匹配到一个时间标签它就停了。 所以我给时间正则加上 g 全局标识，配合 while 循环把当前LCR歌词行所有的时间标签全部取出，同时用正则的 lastIndex 记录一下最后一个标签的位置，方便把后面的歌词文本截取出来。提取完后，遍历所有时间标签，为每个标签匹配一个独立的歌词文本对象。最后用sort对整个数组**按时间升序排序**（因为遍历后时间的顺序可能是乱的），再通过**相邻两行的时间差**算出每行的持续时长 `duration`，方便后面做滚动的高亮匹配。

对于双语歌词匹配，因为翻译歌词其实也是 LRC 格式，所以解析复用了同一parseLRC函数。然后将解析好的翻译歌词数组建成一个Map哈希表，用时间作为key，翻译文本作为 value。 这样遍历原歌词去 Map 里用Map.has查，就能直接匹配到;因为有时存在原歌词时间和翻译歌词时间存在误差的情况，这里做了个容差匹配，这种情况下会遍历整个Map表，找时间差 ≤ 0.5 秒歌词进行组合。
一开始最简单的想法其实是双层 for 循环找，但时间复杂度为 O(n²)。改为哈希表后，查找就变成了 O(1)，整个匹配过程降维成了 O(n) 级别的单层遍历，算是一个小的性能优化点吧。

### 二、二分查找定位当前行（player.ts - `findCurrentLine`）

播放时需要根据当前播放时间快速定位到对应的歌词行。歌词数组已经排好序了，所以我用了**二分查找**。但和普通二分查找不同的是，这里不是精确匹配某个值，而是**区间匹配**——判断当前时间是否落在 `[当前行时间, 下一行时间)` 这个左闭右开区间里。原因很简单：每句歌词都会持续一段时间，不是一个瞬时点。时间复杂度 O(log n)，比线性遍历高效很多。

### 三、requestAnimationFrame 时间驱动（player.ts - `timeLoop`）

requestAnimationFrame是浏览器用于实现高性能动画的 API，它的优势在于执行频率能和浏览器的刷新率保持同步。 具体是，只要音乐在播放，就让requestAnimationFrame的每一帧去跑一个timeloop函数检测循环。
在每一帧里，先拿到 audio 的当前播放时间，然后二分查找算出当前应该高亮哪一行，最后是将这帧算出的行号index和上一帧的index作比较，只有变了才去更新歌词的高亮和平滑滚动。
因为歌词一句可能要唱好几秒，这段时间里如果rAF每帧都去更新 DOM，性能开销会很大。这样就规避了大量无意义的操作。

### 四、GSAP 平滑滚动 + 用户滚动冲突处理（player.ts - `scrollToLine`）

歌词的平滑滚动我用**GSAP** 的 `gsap.to()`来实现。
这里**交互冲突**问题是：如果用户正在手动滚动歌词浏览，自动滚动还在跑的话，页面就会"抢来抢去"。所以我加了一个 `isUserScrolling` 标记，并在歌词容器的最外层监听 `wheel`滚动事件，一旦用户滚动了就把标记设为 `true`，并启动一个 **3 秒的防抖定时器**，自动滚动逻辑看到这个标记就会return放弃接管。3 秒后用户没再滚动，标记恢复为 `false`，自动滚动继续接管。
另外，如果是用户**点击歌词**或者**拖拽进度条**，那就用自动滚动逻辑的`immediate` 模式直接跳转，不受这个标记影响。

### 五、其他细节补充

- **DOM 渲染**用了 `DocumentFragment` 一次性插入，避免逐行 append 导致多次重绘
  const fragment = document.createDocumentFragment()
  this.container.appendChild(fragment)
- **事件委托**：歌词的点击事件只在容器上绑一个，通过 `closest('.lyric-line')` 冒泡找到对应行

## 亮点 2：动态视觉主题与多彩背景

**核心功能**：设计动态视觉主题，基于 ColorThief 从专辑封面提取调色板，通过 RGB→HSL 转换过滤低质量颜色后同步至全局状态，驱动背景渐变与进度条样式联动；同时使用 Canvas 将封面切割生成流动背景，配合双缓冲渐变过渡实现平滑切歌换肤。

### 开场总览

这个功能的核心目标是：每换一首歌，整个应用的视觉风格都会跟着歌曲的封面变化——包括歌曲详情页背景渐变、底部进度条颜色，还有一个流动旋转的背景效果。整体链路是：颜色提取 → 过滤筛选 → 全局广播 → UI 联动。

### 一、颜色提取与过滤（utils/index.ts）

首先通过 ColorThief 这个库从歌曲封面里提取一个调色板（palette），它会返回一组按权重排序的 RGB 颜色。

但直接拿来用是不行的，因为提取出来的颜色可能太黑、太白、太灰，做背景会很难看。所以我做了一轮 RGB → HSL 转换，在 HSL 的饱和度（S）和亮度（L）维度上过滤：亮度低于 0.2 或高于 0.8、饱和度低于 0.2 或高于 0.8 的颜色都会被排除。

过滤后还需要从剩余的颜色里选出最终用于渐变的 2 个颜色。这里我用了一个 **贪心策略（Maximin）**：按权重顺序先选第一个，也就是主色调，然后每次从剩余候选里选一个 **和已选颜色差异最大的**。比较差异时我只比较 S 和 L、忽略色相 H，这样做是为了保证选出来的颜色同色系但有层次变化，比如说选了一个主色是深蓝，第二个颜色就会选择浅蓝，亮蓝色这种同色系但有差异的。

### 二、全局状态同步

选好的颜色通过 Pinia 的 `updateBgColor` 方法存入全局 store，转成 `"89, 134, 167"` 这种字符串格式。ProgressBar 进度条组件直接从 store 里取这个值拼成 `rgb()` 来用，实现了底部播放栏颜色跟着封面走的效果。

### 三、双缓冲渐变过渡（useMusic.ts - `gradualChange`）

切歌时背景渐变需要平滑过渡，不能闪烁。我用的是**双缓冲技术**：页面上叠放了两个完全重合的 div（`#gradual0` 和 `#gradual1`），维护一个 `activeLayerIndex` 指针在 0 和 1 之间切换。

每次切歌时，在透明的那层设置好新渐变，然后把它的 opacity 置为 1，旧层 opacity 置为 0。因为两个 div 都有 `transition: opacity 1s` 的 CSS 过渡，所以视觉上就是一个缓慢的渐变淡入淡出效果。这和显示器的双缓冲原理类似——先在后台画好，再一次性切换。

### 四、Canvas 切割生成流动背景（useMusic.ts - `useRhythm`）

除了纯色渐变，还有一个"节律背景"模式：把歌曲封面用 Canvas 切成 2×2 共 4 块，每块放在容器的四个角，各自以一个随机起始角度做匀速旋转动画（80 秒一圈），再加上一个大的 `filter: blur(80px)` 高斯模糊，最终效果就像一团颜色在缓缓流动。

除了纯色渐变，还有一个"节律背景"模式。核心思路是：用 Canvas 把歌曲封面切成 2×2 共 4 块，通过双重 `for` 循环遍历这个2x2的网格，每次用 `drawImage` 方法从原图对应位置抠出一块画到小 Canvas 上，再用 `toDataURL` 导出为 base64 图片设为 div 的背景。4 个 div 分别定位在容器的四个角，各自通过 CSS 变量 `--start-deg` 设定一个 `Math.random()` 生成的随机起始角度，做 80 秒一圈的匀速旋转动画，再叠加 `filter: blur(80px)` 高斯模糊，最终效果就像一团颜色在缓缓流动。

**性能优化**上做了几件事：

- **Canvas 对象池**：初始化时预创建 4 个 Canvas 存入数组，切歌时直接从池中取出复用，避免反复 `createElement` 带来的内存分配和 GC 压力
- **DOM 复用**：首次运行时 `appendChild` 创建 div，后续切歌通过 `childNodes[index]` 拿到已有节点，只替换 `backgroundImage` 和 `--start-deg`，不销毁重建
- **Canvas 参数优化**：`getContext('2d')` 传入 `alpha: false`（告知浏览器无透明通道，渲染更快）和 `desynchronized: true`（允许异步绘制）；导出图片用 `toDataURL('image/jpeg', 0.8)` 压缩到 80% 质量——反正后面要模糊，损失感知不到，但数据量小很多
- **整个绘图操作包在 `requestAnimationFrame` 里**，确保在浏览器下一帧重绘前执行，不阻塞主线程

### 总结

整条链路就是：ColorThief 提取 → HSL 过滤坏色 → 贪心选最佳组合 → Pinia 广播全局 → 双缓冲渐变过渡 + Canvas 流动背景 → 休眠/唤醒性能兜底。

## 亮点 3：全局音频控制器

**核心功能**：设计全局音频控制器统一管理 Audio 实例，供多个组件共享使用；通过方法劫持扩展原生 Audio API，实现播放/暂停时音量渐变的淡入淡出效果，提升听觉体验并保持播放状态一致性。

### 开场总览

这个功能要解决两个问题：一是多组件共享同一个 Audio 实例，二是播放/暂停时不要生硬地突然响/突然停，而是有音量渐变的淡入淡出效果。

### 一、全局音频控制器的设计

整个应用只有一个 `<audio>` 标签，放在 `MusicPlayer` 组件里。关键问题是：歌词组件要读它的 currentTime、进度条要改它的 currentTime、歌单列表要调它的 play/pause、快捷键模块也要控制播放暂停——这么多地方需要用，怎么统一管理？

我的方案是：

`MusicPlayer` 通过 Vue 的 `defineExpose` 把 audio 元素和控制方法暴露出去，然后在 `App.vue` 的 `onMounted` 里，通过 `useTemplateRef` 拿到组件实例，直接挂到 `window.$audio` 上。这样全站任何地方——不管是 Vue 组件还是纯 TS 工具函数——都可以通过 `window.$audio` 来统一操控播放器。

同时 `isPlay` 是一个 Vue 的 ref，挂到 window 上之后它的响应性依然保留，所以其他组件可以直接 `watch(() => window.$audio.isPlay)` 来监听播放状态变化，不需要额外的事件总线。

### 二、方法劫持实现淡入淡出（Monkey Patching）

淡入淡出的核心是在 `onMounted` 里做了一个方法劫持：

```typescript
// 先备份原生方法
originPlay = audio.value!.play
originPause = audio.value!.pause
// 再覆盖成我们自己的
audio.value!.play = play
audio.value!.pause = pause
```

这样做的好处是：对外的调用方式完全不变，所有调用 `window.$audio.play()` 的地方自动走我们的渐变逻辑，不需要改任何调用方。

渐变的核心是一个 `transitionVolume` 函数，它返回一个 Promise，内部用 `setInterval` 每 50ms 调整一次音量：

- **播放时**：先把 volume 设为 0，调用原生 `originPlay.call(audio)` 真正开始播放，然后从 0 渐增到用户设定的目标音量，分 15 步走
- **暂停时**：从当前音量渐减到 0，分 10 步走。等到真正静音（volume ≤ 0）后，才调用原生 `originPause.call(audio)` 执行物理暂停

这里有个关键细节：暂停时必须用 `originPause.call(audio)` 而不是 `audio.pause()`，否则会递归调用自己导致死循环，因为 `audio.pause` 已经被我们劫持了。

### 三、状态一致性保障

还有几个保证状态一致的设计：

- **`pause` 的 `isNeed` 参数**：切歌时会先暂停当前歌，但切歌暂停不应该把播放图标变成暂停，所以传 `false`；只有用户主动点暂停时才传 `true` 改图标
- **互斥锁 `timeState.stop`**：切歌时先把锁打开，阻止旧 audio 的 `timeupdate` 事件往 store 写数据，防止进度条回弹闪烁
- **`reset` 函数**：切歌时统一重置 currentTime 和锁状态，确保新歌从 0 开始

### 总结

核心就是 `defineExpose + window 挂载` 实现单例全局共享，`方法劫持（Monkey Patching）` 透明地扩展原生 API 加入渐变逻辑，配合 `Promise + setInterval` 的音量步进算法实现平滑过渡，最后通过 `isNeed` 参数和 `stop` 互斥锁保证多场景下的状态一致。

讲下来大概 2-3 分钟。如果面试官追问，方法劫持为什么用 `call` 而不是直接调用和为什么 ref 挂到 window 上还能保持响应性这两个点很容易被展开问。

## 亮点 4：组件休眠/唤醒机制

**核心功能**：针对歌曲详情页设计组件休眠/唤醒机制，页面不可见时降级为轻量渲染并销毁动画实例，避免隐藏组件的无效计算与渲染，后台 CPU 占用由约 15% 降至 <1%，显著提升桌面端运行稳定性。

### 开场：为什么需要这个机制

我们的歌曲详情页是用 `CSS transform: translateY(100%)` 来隐藏的，没有用 `v-if` 销毁组件。原因是这个页面结构很重（Canvas 流动背景、歌词解析器、GSAP 动画），如果每次关闭都销毁、打开都重建，用户体验会很卡。

但问题来了：组件没销毁，`onUnmounted` 永远不会触发，Vue 的 watch 还在跑。用户在不打开歌曲详情页，在后台切歌时，每切一首，watch 依然会触发整套渲染逻辑——加载歌曲、跑 GSAP 缩放动画、Canvas 切割绘图、设置渐变背景。这些操作全是对一个用户看不见的页面做的，完全浪费。实测后台 CPU 占用大约 15%，对桌面应用来说很不友好。

### 核心方案：分级渲染

我的思路是通过一个全局状态 `flags.isOpenDetail` 来区分页面是否可见，同一个 watch 内部做分支判断，实现不同级别的渲染：

**FlowBg.vue（背景渲染）做了两级：**

- **详情页关闭时** → 调用 `executeLightRender`：只做 ColorThief 颜色提取 + Pinia 广播颜色给底部进度条做颜色渐变，不执行 `gradualChange`（双缓冲渐变）和 `splitImg`（Canvas 流动背景）
- **详情页打开时** → 调用 `executeFullRender`：完整执行颜色提取 补上渐变背景和流动背景

**LyricDisplay.vue（封面动画）更激进：**

- **详情页关闭时** → 直接 `currentTimeline.kill()` 销毁 GSAP 时间轴实例，然后 `return`，连图片都不加载
- **详情页打开时** → 正常执行封面缩放动画

```typescript
// 两个组件的核心逻辑结构一样，以 LyricDisplay 为例：
if (!flags.isOpenDetail) {
  // 休眠：杀掉动画，什么都不做
  if (currentTimeline) {
    currentTimeline.kill()
    currentTimeline = null
  }
  return
}
// 正常渲染
executeCoverAnimation(val)
```

### 唤醒机制：补票渲染

休眠之后还有一个问题：用户在后台切了 10 首歌，现在突然打开详情页，界面上还是第 1 首歌的封面和背景，怎么办？

所以两个组件都额外 watch 了 `flags.isOpenDetail`，一旦检测到从 `false` 变成 `true`（页面被打开），就补做一次完整渲染。并且封面动画这边传了 `immediate = true` 参数，直接设置最终状态，跳过缩放动画，避免用户打开时还要等一个收缩-展开的过程。

```typescript
// 唤醒 watcher（FlowBg 和 LyricDisplay 都有类似逻辑）
watch(() => flags.isOpenDetail, (isOpen) => {
  if (isOpen && props.bg) {
    executeFullRender(...) // FlowBg：补做渐变+Canvas
    executeCoverAnimation(props.bg, true) // LyricDisplay：直接贴图，不做动画
  }
})
```

### 效果

优化后，后台切歌时不再执行任何 Canvas 绘图、GSAP 动画、大图加载，CPU 占用从约 15% 降到了 1% 以下。打开详情页时通过唤醒机制补票，用户感知不到任何延迟。本质上就是一个手动实现的组件级 keep-alive 策略——保留 DOM 但冻结计算逻辑。

### 总结

核心思路是 **用全局状态标记可见性 → watch 内分级渲染 + 销毁动画实例 → 唤醒时补做一次渲染**，相当于在不使用 `v-if` 的前提下，手动模拟了组件的"冻结"和"恢复"。

大概 2 分钟讲完。面试官可能会追问：为什么不直接用 `v-if` 或 Vue 的 `<KeepAlive>`——答案就是 `v-if` 销毁重建成本太高（Canvas、歌词解析器等重资源），而 `<KeepAlive>` 虽然缓存了组件但 watch 依然会触发，不能解决后台无效计算的问题，所以需要自己实现这套分级的休眠/唤醒逻辑。
