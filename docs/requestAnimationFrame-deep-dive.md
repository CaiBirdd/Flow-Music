# 深入解析 requestAnimationFrame (rAF)

本文档详细解析了 `requestAnimationFrame` 的运行机制，以及它在歌词播放器 `LyricPlayer` 中的核心作用。

## 1. 为什么会有这个 API？（历史背景）

在 rAF 出现之前（IE9 时代以前），如果你想做一个动画或者循环检测，你只能用 `setTimeout` 或 `setInterval`。

### 笨办法：`setInterval`

```javascript
// 每 16 毫秒执行一次，试图模拟 60帧 (1000ms / 60 ≈ 16.6ms)
setInterval(() => {
  draw() // 更新画面
}, 16)
```

**这种做法有两个致命缺陷**：

1.  **瞎忙活 (不精准)**：`setInterval` 是不管死活的。就算浏览器正在忙着处理 1个亿的计算，或者此时屏幕还没刷新完，它依然会强行把任务塞进队列。导致动画卡顿、丢帧、抖动。
2.  **不休息 (费电)**：当你把浏览器最小化，或者切到别的标签页看新闻时，这个 `setInterval` **依然在后台疯狂运行**，每秒 60 次，导致笔记本风扇狂转，电量尿崩。

### 救世主：`requestAnimationFrame`

浏览器厂商推出的 rAF，其核心逻辑是：**“原本由 JS 决定什么时候画，现在改由 屏幕/浏览器 决定什么时候画”。**

---

## 2. rAF 的核心特性

### 1. 跟着刷新率走 (VSync 同步)

- 即便你写了无限循环，rAF 也是有节操的。
- 如果是 60Hz 屏幕，它每秒跑 60 次。
- 如果是 144Hz 电竞屏，它每秒跑 144 次。
- 它保证了 JS 每一帧只执行一次，且正好在**屏幕下一次刷新之前**执行。这让动画如丝般顺滑。

### 2. 智能休眠 (Power Saving)

- **这是重点优化！** 当你切到别的标签页，或者把浏览器最小化时，**rAF 会自动暂停！**
- JS 引擎完全停止工作，CPU 占用率直接降为 0。
- 一旦你切回来，它立刻从暂停的地方继续跑。这就是为什么我们在性能优化文档中提到配合 rAF 可以进一步降低后台开销。

## 3. 结合业务代码分析

在 `player.ts` 中，我们的 `timeLoop` 是标准的 **rAF 递归调用模式**（准确说是**调度模式**）：

```typescript
private timeLoop = (): void => {
  // A. 执行业务逻辑 (查岗、高亮歌词)
  // 如果当前行号变了，才操作 DOM
  if (index !== this.currentIndex) {
     this.updateLine(index)
  }

  // B. 预约下一帧
  // 这句话的意思是："浏览器大哥，等你下一帧准备刷新画面的时候，麻烦再叫我(this.timeLoop)一次"
  this.rafId = requestAnimationFrame(this.timeLoop)
}
```

### 执行流程图解：

1.  **第 0ms**：调用 `play()` -> 触发 `startLoop()` -> 触发第一次 `rAF(timeLoop)`。
2.  **浏览器的内心戏**：“收到！但我现在正忙着显示这一帧呢。等我把这一帧画完，准备画下一帧的时候叫你。”
3.  **第 16.6ms (屏幕准备好了)**：浏览器唤醒 `timeLoop` 函数。
    - JS 醒来，检查时间，发现该换歌词了，更新 DOM。
    - 代码最后一行又执行了 `rAF(timeLoop)`，再次预约。
4.  **第 33.2ms... 第 49.8ms...** 如此往复。

### 递归 vs 调度：它到底是不是递归？

这是一个容易引起混淆的概念，我们需要从**逻辑**和**物理（内存）**两个层面来区分：

1.  **逻辑层面**：它确实是**“异步递归”**。
    函数内部引用了自身，并在未来某个时间点再次执行自己。这种“自己调用自己”的模式，在算法逻辑上构成了递归。

2.  **物理层面（内存栈）**：它**绝对不是递归**。
    - **同步递归（True Recursion）就像搭积木**：函数 A 调用 B，A 必须等 B 结束才能结束。调用层级越深，内存栈压得越高，最终会导致“栈溢出 (Stack Overflow)”。
    - **rAF 调度（Scheduling）就像接力跑**：
      1.  第一棒 `timeLoop` 跑完 -> **函数结束，销毁出栈** -> 内存清空。
      2.  浏览器等待 16ms。
      3.  浏览器取出第二棒 `timeLoop` 重新压栈执行 -> **函数结束，销毁出栈**。

**结论**：严谨地说，这是一个**基于事件循环的异步调度循环**。它拥有递归的逻辑结构，但完全没有递归的内存开销风险。

## 4. 生命周期管理：`rafId`

```typescript
// 启动
this.rafId = requestAnimationFrame(this.timeLoop)

// 停止
cancelAnimationFrame(this.rafId)
```

**比喻：拿着挂号单**

- 当你调用 `requestAnimationFrame` 时，浏览器会给你一张**挂号单 (ID，一个整数)**。
- 如果你突然不想看了（用户点了暂停 `pause()`，或者切歌了），你必须拿着这张单子去护士台（`cancelAnimationFrame`）销号。
- **重要**：在 `startLoop` 中，我们采取了防御性编程，先 `cancel` 掉上一个可能存在的 ID 再重新申请，防止“由于连点播放按钮导致的 N 倍速循环”。

---

## 总结

在歌词播放器中使用 rAF 替代 `setInterval` 带来了以下收益：

1.  **视觉流畅**：歌词滚动与屏幕刷新率完美对齐。
2.  **性能极佳**：后台自动暂停，不浪费每一度电。
3.  **防抖动**：浏览器自动协调任务队列，避免 Layout Thrashing。
