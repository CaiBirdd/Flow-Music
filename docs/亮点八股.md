# 面试项目亮点：关联“八股”题库及精简回答

这份文档结合了项目的四大亮点，梳理了面试官可能顺着亮点追问的基础知识（八股文）及其精简回答。

## 亮点 1：LRC 歌词解析与播放

### 1. 正则表达式：`exec` 与 `match` 的区别？`g` 标志和 `lastIndex` 的作用？

**答案：**

- **无 `g` 标志时：** 两者行为几乎一样，都返回第一个匹配结果及其捕获组详情（groups、index）。
- **有 `g` 标志时（关键区别）：**
  - `match` 变成"批量模式"——一次性返回**所有匹配子串的数组**，但**丢失捕获组和 index**，只拿到匹配到的文本。
  - `exec` 仍然是"逐次模式"——每次调用只返回**一个匹配结果（含捕获组）**，同时更新正则对象的 `lastIndex`（下次匹配的起始位置），需要 `while(exec)` 循环逐个取出。
- **项目选择 `exec` 的原因：** 歌词解析既需要捕获组提取分、秒、毫秒，又需要 `lastIndex` 定位最后一个时间标签的结束位置来截取歌词文本，这两个能力只有 `exec` 同时具备。
- **为什么不用 `matchAll`？** `matchAll` 是 ES2020 新增的，返回一个迭代器，能拿到所有匹配的捕获组详情，功能上可以替代 `while(exec)` 循环。项目里用 `exec` 是因为除了提取时间标签，还需要 `lastIndex` 来定位最后一个标签的结束位置以截取歌词文本，`matchAll` 不提供这个能力。

### 2. 正则：贪婪匹配和非贪婪匹配的区别？

**答案：**

- **贪婪模式（默认）：** 量词（`*`、`+`、`?`、`{n,m}`）会尽可能多地匹配字符。如 `/a.+b/` 匹配 `"aXbYb"` 时会匹配整个 `"aXbYb"`（贪心到最后一个 b）。
- **非贪婪模式：** 在量词后加 `?`（如 `*?`、`+?`），尽可能少地匹配。`/a.+?b/` 匹配 `"aXbYb"` 时只匹配 `"aXb"`（遇到第一个 b 就停）。
- **实际应用：** 比如要匹配 HTML 标签 `<div>content</div>` 中的内容，贪婪的 `/<.+>/` 会匹配整个字符串，非贪婪的 `/<.+?>/` 才能正确匹配单个标签 `<div>`。

### 3. 算法：为什么要用二分查找而不是遍历？标准二分和当前业务"区间匹配"区别在哪？

**答案：**

- LRC 歌词数组是按时间严格排好序的，用二分查找时间复杂度 O(log n)，比从头遍历效率高得多。
- **为什么是 O(log n)：** 每次比较都能排除一半的候选元素，n 个元素最多需要 log₂n 次比较就能找到目标。
- 标准二分是找绝对相等的目标值，而当前的业务是**"寻找最后一个时间小于等于当前播放时间的歌词行"**（或者说落在 `[当前行, 下一行)` 区间内）。
- **边界条件 `left < right` vs `left <= right`：** `left <= right` 用于精确查找，循环结束时 `left > right` 表示未找到；`left < right` 常用于查找边界（如"最后一个 ≤ target 的位置"），循环结束时 `left === right` 收敛到答案。歌词区间匹配用的是 `left < right` 变体，因为我们要找的不是精确值，而是"满足 ≤ 当前播放时间的最后一个位置"。

### 4. 动画控制：`requestAnimationFrame` (rAF) 是什么？为何优于 `setInterval`？

**答案：**

- rAF 是浏览器专门为动画提供的 API。它的回调执行时机**与屏幕刷新率同步**（通常 16.7ms / 60Hz），在事件循环（Event Loop）的**渲染阶段之前**执行，不会由于主线程阻塞造成掉帧。
- 另外，**当页面不可见或处于后台标签页时，rAF 会自动暂停**，这极大节省了 CPU 和电池资源，而 `setInterval` 仍在盲跑。

### 5. DOM 优化：`DocumentFragment` 和事件委托的原理？

**答案：**

- **`DocumentFragment`：** 是内存中的一段虚拟 DOM 节点。逐次 append 会引发多次页面重排/重绘，把所有节点先塞进 Fragment 再一次性挂载，只触发一次渲染流水线。
- **事件委托：** 利用**事件冒泡**机制，不给每行歌词绑事件，而是全绑在外层容器上。通过判断 `event.target` 定位点击的是哪行，不仅节省内存，对后续动态添加的节点依然有效。
- **冒泡和捕获的区别：** DOM 事件流分三个阶段——**捕获阶段**（从 window → document → ... → 目标父元素）→ **目标阶段** → **冒泡阶段**（从目标元素 → ... → document → window）。`addEventListener` 的第三个参数为 `true` 时在捕获阶段触发，默认 `false` 在冒泡阶段触发。事件委托利用的是冒泡阶段。
- **`event.target` vs `event.currentTarget`：** `target` 是实际触发事件的最内层元素（如被点击的那行歌词），`currentTarget` 是事件监听器绑定的元素（如外层容器）。事件委托正是利用了二者的差异。

### 6. 事件循环（Event Loop）：rAF 在事件循环中的执行时机是什么？

**答案：**

完整流程：**宏任务（如 setTimeout 回调）→ 清空微任务队列（Promise.then、MutationObserver）→ 判断是否需要渲染 → 执行 rAF 回调 → 执行渲染（Layout → Paint → Composite）→ 空闲时执行 requestIdleCallback**。

rAF 处于"渲染阶段之前"，所以它能拿到最新的 DOM 状态并在同一帧内生效。而 `setTimeout/setInterval` 属于宏任务，可能和渲染不同步导致掉帧。

### 7. 重绘（Repaint）和回流（Reflow）的区别？

**答案：**

- **回流（Reflow / Layout）：** 元素的几何属性（位置、尺寸）发生变化时，浏览器需要重新计算布局，代价最高。触发场景：改 `width`/`height`/`margin`/`padding`、读取 `offsetWidth`/`getBoundingClientRect`、增删 DOM 节点等。
- **重绘（Repaint）：** 仅外观变化（如 `color`、`background-color`），不影响布局，代价较低。
- **回流一定引发重绘，重绘不一定引发回流。** 性能优化的核心思路就是尽量减少回流。

### 8. 防抖（Debounce）和节流（Throttle）的区别？

**答案：**

- **防抖：** 事件触发后等待一段时间才执行，如果等待期间又触发则重新计时。适合"结束后再做"的场景（如搜索输入、用户停止滚动后恢复自动滚动）。项目中歌词的用户滚动冲突处理用的就是 3 秒防抖。
- **节流：** 固定时间间隔内最多执行一次，不管触发多少次。适合"持续触发但限频"的场景（如滚动事件、resize 事件）。
- 核心区别：防抖是"只认最后一次"，节流是"固定频率执行"。

### 9. `Map` 和普通对象 `{}` 的区别？为什么双语歌词匹配用 Map？

**答案：**

- **键的类型：** 对象的键只能是 `string` 或 `Symbol`，Map 的键可以是任意类型（包括数字、对象等）。歌词时间是数字，用 Map 不需要做隐式类型转换。
- **性能：** Map 在频繁增删查改的场景下性能更优，底层是哈希表实现，`has/get` 的平均时间复杂度是 O(1)。
- **有序性：** Map 保持插入顺序，对象在数字键时会自动排序。
- **大小获取：** Map 有 `.size` 属性，对象需要 `Object.keys().length`。

### 10. `Array.prototype.sort` 的底层实现是什么？

**答案：**

- V8 引擎使用 **TimSort**（Python 发明的混合排序算法），结合了归并排序和插入排序的优点。对于小数组用插入排序（常数因子小），大数组用归并排序（稳定且 O(n log n)）。
- 注意：`sort()` 默认按字符串 Unicode 排序，所以对数字排序必须传比较函数，如 `sort((a, b) => a.time - b.time)`。

## 亮点 2：动态视觉主题与多彩背景

### 1. 色彩空间：为什么在提取颜色后，要转为 HSL 进行过滤而不是直接用 RGB？

**答案：**

- RGB 是红绿蓝发光模型，它是给显示器看的，不符合人类的感官直觉。
- **HSL（色相、饱和度、亮度）更贴近人眼认知。** 项目中需要剔除太暗、太白、太灰的颜色，直接在 HSL 维度的 `S`（饱和度）和 `L`（亮度）上设阈值（如排除 `<0.2` 或 `>0.8`）非常精准直观。
- **RGB → HSL 转换思路：** 将 R、G、B 归一化到 [0,1]，先由 max/min 求亮度 **L = (max+min)/2**，再由 max-min 差值求饱和度 **S**，最后根据最大分量是 R/G/B 分三种情况算色相 **H**（0-360°）。核心是从"亮不亮"→"鲜不鲜"→"什么颜色"逐步推导。

### 2. Canvas 性能调优：你提到了哪些关于 Canvas 的性能优化细节？

**答案：**

- **关闭透明度 (`alpha: false`)：** 明确告诉浏览器画布非透明，跳过透明度混合带来的计算开销。
- **异步绘制 (`desynchronized: true`)：** 允许 Canvas 绑定的绘制操作脱离主事件循环，减少延迟。这在低延迟绘图（如手写板）场景更有意义，项目中用它是为了尽量减少绘图对主线程的阻塞。
- **对象池（Object Pool）：** 初始化时创建好几个 Canvas 对象备用，切歌时直接复用，避免高频 `createElement` 触发浏览器垃圾回收（GC）导致卡顿。
- **降质导出 (`toDataURL` 压缩)：** 最终呈现是被 `filter: blur` 严重模糊的，导出 base64 时牺牲画质只保留 20-80% 质量，能显著减小体积。

### 3. CSS 动画底座：触发双缓冲用到了 `transition: opacity` 和 `transform`，从浏览器渲染机制解释为什么性能好？

**答案：**

- 浏览器的渲染流程包括：**Layout（重排/布局） → Paint（重绘） → Composite（合成）**。
- `width`/`height` 改变会触发最消耗性能的重排。而修改 `opacity` 或 `transform`（如 translate、rotate），结合硬件加速，会**跳过重排与重绘，直接在 GPU 触发合成层（Composite layer）的变化**。双缓冲利用两层重叠的 DOM 错开切换透明度，不会引发整个页面的布局重排。

### 4. 算法思想：贪心算法的适用条件？和动态规划的区别？

**答案：**

- **贪心算法：** 每一步都选当前看起来最优的选项，不回溯。适用于具有**贪心选择性质**（局部最优能推导出全局最优）和**最优子结构**的问题。项目中选颜色时每次从候选里选差异最大的，就是贪心策略。
- **动态规划（DP）：** 通过拆分子问题并记忆化存储，穷举所有可能，保证全局最优。
- **核心区别：** 贪心不回头、效率高但不保证全局最优；DP 考虑所有情况、保证全局最优但时间/空间开销更大。选颜色的场景用贪心已经足够好，不需要 DP 那样穷举所有组合。

### 5. 双缓冲（Double Buffering）的原理？在前端还有哪些应用场景？

**答案：**

- **原理：** 维护两个缓冲区（如两个 div），一个显示当前内容（前台），一个在后台准备新内容。切换时把后台推到前台，原来的前台变后台等待下次更新。避免"画到一半被用户看到"的撕裂/闪烁。
- **前端场景：** Canvas 游戏的离屏渲染（先画到离屏 Canvas 再一次性拷贝到主 Canvas）、轮播图切换、骨架屏到真实内容的过渡等。

### 6. `toDataURL` 和 `toBlob` 的区别？Base64 编码的原理？

**答案：**

- **`toDataURL`：** 同步方法，返回 base64 编码的字符串（`data:image/jpeg;base64,...`），可以直接赋值给 `src` 或 `backgroundImage`，简单方便但会阻塞主线程。
- **`toBlob`：** 异步方法，回调返回 Blob 对象（二进制数据），不阻塞主线程，适合大图或需要上传的场景。
- **Base64 原理：** 用 64 个可打印 ASCII 字符（A-Z、a-z、0-9、+、/）来表示二进制数据。每 3 个字节（24 bit）编码为 4 个字符（每个 6 bit），所以体积会膨胀约 **33%**。

### 7. `filter: blur()` 为什么"贵"？有什么优化手段？

**答案：**

- **开销来源：** 模糊需要对每个像素采样周围一定半径内的所有像素做加权平均（高斯卷积），半径越大计算量越大，是 O(n × r²) 级别的操作。
- **优化手段：**
  - 使用 `will-change: filter` 或 `transform: translateZ(0)` 提升为独立合成层，让 GPU 负责模糊计算。
  - 降低被模糊元素的实际分辨率（如项目中用 80% 质量导出），反正模糊后看不出差别。
  - 避免对大面积、高分辨率元素做实时模糊。

### 8. GPU 加速与 `will-change` 属性？

**答案：**

- 浏览器默认在 CPU 上执行渲染，但 `transform`、`opacity`、`filter` 等属性的变化可以被提升到 **GPU 合成层**，由显卡独立处理，不占主线程。
- **`will-change`：** 提前告诉浏览器某个属性即将变化，让它预先创建合成层。比如 `will-change: transform` 会让元素提前提升到 GPU 层。
- **滥用风险：** 每个合成层都占显存，过多使用会导致内存飙升甚至卡顿。应该只对确实会做动画的元素使用，动画结束后可以移除。

### 9. 状态管理：Pinia 和 Vuex 的区别？

**答案：**

- **API 设计：** Pinia 去掉了 Vuex 的 `mutations`（只有 `state`、`getters`、`actions`），代码更简洁直接。
- **TypeScript 支持：** Pinia 天生对 TS 友好，类型推导完整；Vuex 的 TS 支持较弱。
- **模块化：** Pinia 每个 store 是独立定义的，不需要 Vuex 的 `modules` 嵌套结构。
- **响应性底层：** 两者都基于 Vue 的响应式系统（Vue 3 下是 `reactive`/`ref`），本质一样，只是 API 封装不同。
- **体积：** Pinia 更轻量（约 1KB gzipped），是 Vue 官方推荐的新一代状态管理方案。

### 10. Canvas 绘图为什么要放在 rAF 里？不放会怎样？

**答案：**

- Canvas 的 `drawImage`、`toDataURL` 等操作是同步阻塞主线程的。如果直接在 `watch` 回调中执行，同步代码会立即执行，可能在一帧内堆积多次重复绘制（比如多个响应式数据连续变化），造成掉帧。
- 放在 `requestAnimationFrame` 里有两个好处：① 多次触发会被**合并到同一帧**执行（rAF 回调每帧只执行一次），天然起到节流作用；② 绘图时机与浏览器渲染节奏对齐，确保画面在下一帧完整呈现，避免撕裂。

### 11. 对象池模式（Object Pool）的原理和适用场景？和享元模式的区别？

**答案：**

- **对象池：** 预先创建一批对象放在"池"里，需要时取出使用，用完放回而不是销毁。适合**创建/销毁成本高且频繁使用**的场景，如 Canvas 元素、数据库连接、线程等。项目中预创建 4 个 Canvas 避免反复 `createElement`。
- **享元模式（Flyweight）：** 侧重于**共享不可变的内在状态**，把可变的外在状态提取到外部。比如 1000 个相同样式的粒子共享同一个样式对象。
- **核心区别：** 对象池里的对象是**独占使用、用完归还**的；享元模式的对象是**同时被多方共享**的。

### 12. CSS `transition` 和 `animation` 的区别？

**答案：**

- **`transition`：** 需要一个触发条件（如 `:hover`、class 切换、JS 修改样式），只能定义**起始状态和结束状态**之间的过渡，执行一次。适合简单的状态变化，如双缓冲的 opacity 切换。
- **`animation`：** 通过 `@keyframes` 定义多个关键帧，可以自动播放、无限循环、暂停恢复、正反交替。适合复杂的、持续性的动画，如流动背景的旋转动画。
- **性能：** 两者本身性能差异不大，关键在于动画的属性——`transform` 和 `opacity` 只触发合成，性能最好；`width`/`top` 等触发回流，性能最差。

## 亮点 3：全局音频控制器

### 1. 软件设计模式：这算是单例模式吗？除了挂载全局对象，前端还有什么方式实现单例？

**答案：**

- 算。全局仅暴露及维护唯一一个 Audio 控制器，保障多组件统一状态。
- 如果不使用 `window` 挂载，现代前端常使用 **ES Module 的导出**（模块在第一次加载被解析执行后结果会被缓存，导出的实例天然是单例），或使用状态库工具如 Pinia、Redux 维护单实例状态。

### 2. JS 特性应用：方法劫持 (Monkey Patching) 的原理是什么？结合你使用 `originPause.call(audio)` 说明为什么不用直接调用？

**答案：**

- **方法劫持：** 是指在运行时将目标对象的原生方法替换为我们自己的包装函数（先加自定义逻辑再调用原方法）。
- **为什么不能 `audio.pause()`：** 因为 `audio.pause` 已经被我们覆盖成了自定义的淡出函数。如果在自定义函数内部再调 `audio.pause()`，就等于调用自己 → 自己又调自己 → 无限递归 → 调用栈溢出（Stack Overflow）。所以我们必须事先把原生方法备份到 `originPause` 变量里，在包装函数内部调用这个备份。
- **为什么要用 `call`：** `originPause` 只是一个独立的函数引用，已经脱离了 `audio` 对象。直接 `originPause()` 调用时，`this` 会指向 `undefined`（严格模式）或 `window`，而浏览器原生的 `pause()` 方法内部需要 `this` 指向正确的 `HTMLAudioElement` 才能工作。所以必须用 `originPause.call(audio)` 手动把 `this` 绑回 audio 元素。
- **Monkey Patching 的优缺点：** 优点是对调用方完全透明，不需要改任何外部代码；缺点是侵入性强，调试困难（断点会进到包装函数），且如果多个库都 patch 同一个方法会互相覆盖。
- **更优雅的替代方案——Proxy：** ES6 的 `Proxy` 可以通过 `handler.get`/`handler.apply` 等 trap 拦截对象操作，不修改原始对象。但 `Proxy` 无法直接代理 DOM 原生对象的方法调用（浏览器内部方法要求正确的 `this`），所以本项目中 Monkey Patching 是更务实的选择。

### 3. 异步方案：音量淡入淡出的 `transitionVolume` 返回 Promise，如何控制 resolve 时机？

**答案：**

- 在 `new Promise` 的执行器中开启 `setInterval` 逐步修改音量。只有当音量达到目标值并执行了真正的物理播放/暂停后，才在该条件分支内手动调用 `resolve()`。外部 `await transitionVolume(...)` 会**挂起当前 async 函数**（不阻塞主线程），直到 `resolve` 被调用才继续往下执行。

### 4. Vue 底层：将内部 Vue 的响应式参数（如 `isPlay` ref）暴露到 Window 上，响应性会丢失吗？为什么？

**答案：**

- 不会丢失。Vue 3 的响应式核心是利用 **Proxy** 或提供内部 `getter/setter` 拦截的对象（RefImpl）。`ref` 生成的响应式对象本质还是那个被拦截的对象引用，不管被转移传递或赋值到哪个层级（包括 window），它自身的拦截器机制都在。
- **Vue 3 Proxy 响应性原理：** `reactive()` 内部用 `new Proxy(target, handler)` 包裹原始对象，在 `handler.get` 中做**依赖收集**（track，将当前正在运行的 effect/computed/watch 记录到该属性的订阅者集合），在 `handler.set` 中做**派发更新**（trigger，通知所有订阅者重新执行）。`ref()` 则是封装了一个带 `.value` 的 `getter/setter` 对象（RefImpl），访问 `.value` 时 track，修改 `.value` 时 trigger。

### 5. `call`、`apply`、`bind` 的区别？

**答案：**

- **`call(thisArg, arg1, arg2, ...)`：** 立即调用函数，参数逐个传入。
- **`apply(thisArg, [args])`：** 立即调用函数，参数以数组传入。
- **`bind(thisArg, arg1, ...)`：** **不立即调用**，返回一个永久绑定 `this` 的新函数。
- 项目中用 `originPause.call(audio)` 而不是 `bind` 是因为需要立即执行，不需要保存一个新函数引用。

### 6. `this` 指向的完整规则？

**答案：**

按优先级从高到低：

1. **`new` 绑定：** `new Foo()` 中 `this` 指向新创建的对象。
2. **显式绑定：** `call`/`apply`/`bind` 手动指定 `this`。
3. **隐式绑定：** `obj.fn()` 中 `this` 指向调用者 `obj`。
4. **默认绑定：** 独立调用 `fn()`，非严格模式指向 `window`，严格模式为 `undefined`。
5. **箭头函数：** 没有自己的 `this`，继承定义时外层作用域的 `this`，且无法被 `call/bind` 改变。

### 7. 闭包（Closure）是什么？会导致内存泄漏吗？

**答案：**

- **定义：** 函数能够访问其定义时所在的词法作用域中的变量，即使该函数在其他地方被调用。`transitionVolume` 内部的 `setInterval` 回调就是一个闭包——它引用了外层函数的 `resolve`、`targetVolume` 等变量。
- **内存泄漏：** 闭包本身不会泄漏，但如果闭包引用了大对象且闭包本身长期不释放（如被全局变量持有、未清除的定时器），那被引用的对象也无法被 GC 回收，这才构成泄漏。项目中在音量达到目标值后 `clearInterval` 就是为了释放闭包。

### 8. `setInterval` 的缺陷是什么？

**答案：**

- **不精确：** `setInterval(fn, 50)` 不是保证每 50ms 执行一次，而是每 50ms 将回调**加入宏任务队列**。如果主线程被阻塞，实际执行间隔会更长。
- **堆积问题：** 如果回调执行时间超过间隔时间，多个回调会在队列中堆积，导致连续快速执行。
- **替代方案：** 用递归 `setTimeout` 可以保证每次回调执行完后再等待固定时间；动画场景用 `requestAnimationFrame`。

### 9. `defineExpose` 的作用？Vue 3 的 `<script setup>` 为什么默认封闭？

**答案：**

- `<script setup>` 是 Vue 3 的编译时语法糖，默认**不对外暴露任何内部变量和方法**，组件实例上拿不到。这是为了更好的封装性和 Tree-shaking（未暴露的变量可以被编译器优化）。
- `defineExpose({ ... })` 显式声明哪些属性/方法可被父组件通过 `ref` 访问。项目中 `MusicPlayer` 通过它暴露 `audio` 元素和控制方法，使得 `App.vue` 能拿到并挂载到 `window.$audio`。

### 10. Promise 的三种状态？`Promise.all` 和 `Promise.race` 的区别？

**答案：**

- **三种状态：** `pending`（进行中）→ `fulfilled`（已成功）或 `rejected`（已失败）。状态一旦改变就不可逆（settled）。
- **`Promise.all([p1, p2, ...])`：** 所有 Promise **全部成功**才 resolve（返回结果数组），任意一个 reject 就立即 reject。适合"所有任务都要完成"的场景。
- **`Promise.race([p1, p2, ...])`：** **最先**改变状态（无论成功还是失败）的那个 Promise 决定结果。适合"超时控制"场景（竞赛一个请求和一个 setTimeout）。
- **`Promise.allSettled`：** 等所有 Promise 都 settled（不管成功失败），返回每个的状态和结果，不会短路。

### 11. 发布订阅模式 vs 观察者模式的区别？Vue 的 `watch` 属于哪种？

**答案：**

- **观察者模式（Observer）：** 目标对象（Subject）直接维护一个观察者列表，状态变化时直接通知观察者。Subject 和 Observer **互相知道对方的存在**。
- **发布订阅模式（Pub/Sub）：** 发布者和订阅者之间有一个**中间调度中心（Event Bus）**，发布者只管发事件，订阅者只管订阅事件，双方互不认识，完全解耦。
- **Vue 的 `watch`** 更接近**观察者模式**：响应式数据（Subject）通过 Proxy 的 getter 收集依赖（Observer），数据变化时通过 setter 直接通知依赖执行，中间没有事件总线。

### 12. `useTemplateRef` 获取组件实例的原理？和 `$refs` 的区别？

**答案：**

- **`useTemplateRef`（Vue 3.5+）：** 传入一个字符串 key，Vue 在模板编译时自动将匹配 `ref="key"` 的 DOM/组件实例绑定到返回的 ref 上。本质是 `ref(null)` + 编译时自动赋值。
- **传统 `$refs`：** 是 Options API 的产物，通过 `this.$refs.xxx` 访问。在 `<script setup>` 中不能直接用 `$refs`，取而代之的是 `useTemplateRef` 或直接声明同名的 `ref` 变量。
- **核心区别：** `useTemplateRef` 返回一个响应式 ref，可以 `watch` 它的变化；`$refs` 是非响应式的普通对象，不能 watch。

## 亮点 4：组件休眠与唤醒机制

### 1. 生命周期与内置组件：既然隐藏开销大，为什么不能用 Vue 官方的 `<KeepAlive>` 处理？

**答案：**

- `<KeepAlive>` 的原理是在组件失活时（触发 `deactivated` 钩子）把 VNode 及实例状态缓存到内存，激活时（`activated`）取出复用，避免组件从 0 挂载。
- **核心痛点：** `<KeepAlive>` 的设计目标是**保持组件完全存活**——包括 DOM 状态和所有 `watch` 监听器。这恰恰是我们不想要的：后台切歌时 `watch` 仍会响应数据变化，重跑繁重的 Canvas 和计算逻辑。虽然可以在 `deactivated` 钩子里手动停止逻辑，但我们的详情页不是路由组件、不走 `<router-view>`，无法被 `<KeepAlive>` 包裹，所以必须自建基于 `isOpen` 全局状态的"业务执行拦截器"。

### 2. 浏览器机制对比：你项目里的 `transform: translateY(100%)` 和常见的 `display: none` / `v-if` 对引擎的区别？

**答案：**

- `v-if`：从组件到 DOM 节点，全部物理彻底销毁和析构。
- `v-show` / `display: none`：元素存在于 DOM 树，但从渲染树（Render Tree）剔除，不占空间，切换会引起**重排（Reflow）**。
- `transform / opacity: 0`：不仅存在 DOM 树也存在渲染树，保留原来的占位，改变它们仅在 GPU 层进行矩阵变换，即只触发**合成（Composite）**，性能最优。
- **为什么不用 `v-show`：** `v-show` 底层是 `display: none`，虽然不销毁组件但切换时会触发重排。而 `transform: translateY(100%)` 只触发合成，并且元素仍然在渲染树中，切回来时不需要重新构建渲染树，体验更流畅。

### 3. 系统级优化思想：这体现了怎样的主流前端优化思路？

**答案：**

- 这种在不可见时降级或挂起的思想，和前端流行的 **虚拟列表（Virtual List）** 以及基于 **`IntersectionObserver`（可见性侦测）懒加载** 的本质是一样的，即：**优先渲染可见视区，冻结或忽略视区外物体的无效计算，把宝贵的 CPU 时间留给主线程**。

### 4. Vue 的完整生命周期？

**答案：**

- **创建阶段：** `setup()`（Composition API 入口）→ `onBeforeMount`（DOM 挂载前）→ `onMounted`（DOM 挂载完成，可操作真实 DOM）
- **更新阶段：** `onBeforeUpdate`（响应式数据变化、DOM 更新前）→ `onUpdated`（DOM 更新完成）
- **卸载阶段：** `onBeforeUnmount`（卸载前，清理定时器/事件监听）→ `onUnmounted`（完全卸载）
- **KeepAlive 专属：** `onActivated`（缓存组件激活）→ `onDeactivated`（缓存组件失活）
- 项目中详情页不走 `onUnmounted`（因为没用 `v-if`），这正是需要手动实现休眠机制的原因。

### 5. `watch` 和 `watchEffect` 的区别？

**答案：**

- **`watch`：** 需要显式指定监听的数据源，回调中能拿到新值和旧值，默认是懒执行的（第一次不触发，除非设 `immediate: true`）。
- **`watchEffect`：** 不需要指定数据源，自动收集回调中用到的所有响应式依赖，**创建时立即执行一次**。
- **`watch` 的 `deep` 选项：** 默认 `watch` 只监听引用变化（浅层），设 `deep: true` 会递归遍历对象所有嵌套属性进行深度监听。注意深度监听有性能开销，大对象慎用。
- **选择依据：** 需要知道新旧值对比或精确控制触发时机用 `watch`；只关心"依赖变了就重新执行"用 `watchEffect`。
- 项目中监听歌曲切换用 `watch` 是因为需要 `immediate` 控制和明确的数据源。

### 6. `nextTick` 的原理是什么？

**答案：**

- Vue 的 DOM 更新是**异步批量**执行的。当修改响应式数据后，Vue 不会立即更新 DOM，而是把更新任务推入一个队列，在当前同步代码执行完毕后（利用微任务 `Promise.then`）统一刷新。
- `nextTick()` 返回一个 Promise，它的回调会在 DOM 更新完成后执行。本质就是往微任务队列里插一个回调，排在 Vue 的 DOM 更新之后。
- 使用场景：修改数据后需要立即读取更新后的 DOM（如获取元素尺寸、滚动位置）。

### 7. 内存泄漏的常见场景？

**答案：**

- **未清除的定时器：** `setInterval`/`setTimeout` 没有 `clearInterval`/`clearTimeout`，回调的闭包持续持有外部引用。
- **未移除的事件监听：** `addEventListener` 后组件销毁时没有 `removeEventListener`。
- **未销毁的第三方实例：** 如 GSAP 时间轴、Chart.js 图表、ECharts 实例等，不调用 `kill()`/`destroy()` 会一直驻留内存。项目中详情页关闭时 `currentTimeline.kill()` 就是为了防止这个问题。
- **闭包意外引用：** 闭包捕获了大对象但闭包本身长期存活。
- **脱离 DOM 的引用：** JS 变量仍引用已从 DOM 树移除的元素，导致元素无法被 GC。

### 8. Chrome DevTools 的 Performance 面板怎么用？CPU 占用是怎么测量的？

**答案：**

- **录制：** 打开 Performance 面板，点 Record，操作页面（如后台切歌），然后 Stop。面板会展示一条时间线，包含 **CPU 火焰图**（调用栈）、**帧率**（FPS）、**内存占用**等。
- **分析 CPU：** 时间线底部的 **Summary** 饼图会显示 Scripting（JS 执行）、Rendering（布局/重排）、Painting（绘制）各占比。如果后台切歌时 Scripting 占比很高，说明有无效的 JS 计算。
- **Electron 场景：** 还可以打开系统的任务管理器直接观察渲染进程的 CPU 占用百分比，对比优化前后的数值变化。

### 9. 浏览器的完整渲染流程？

**答案：**

1. **构建 DOM 树：** 解析 HTML，生成 DOM Tree。
2. **构建 CSSOM 树：** 解析 CSS，生成 CSSOM Tree。
3. **合并为渲染树（Render Tree）：** DOM + CSSOM 合并，排除不可见元素（如 `display: none`）。
4. **Layout（布局/回流）：** 计算每个节点的精确位置和尺寸。
5. **Paint（绘制）：** 将节点的视觉属性（颜色、阴影等）绘制到图层上。
6. **Composite（合成）：** 将多个图层按顺序合成为最终画面，由 GPU 负责。

修改 `transform`/`opacity` 之所以快，就是因为跳过了 Layout 和 Paint，直接在 Composite 阶段处理。

### 10. CSS `contain` 属性了解吗？可以怎么优化你的隐藏场景？

**答案：**

- **`contain`** 告诉浏览器某个元素的内部变化**不会影响外部布局**，浏览器可以做更激进的优化（减少回流范围）。
- 常用值：
  - `contain: layout`：元素内部布局变化不影响外部。
  - `contain: paint`：元素内部绘制不会溢出边界。
  - `contain: size`：元素大小不依赖子元素（需显式设尺寸）。
  - `contain: strict`（= `size layout paint`）：最强隔离。
  - `contain: content`（= `layout paint`）：常用的折中方案。
- 对于详情页隐藏场景，可以给隐藏的容器加 `contain: strict`，让浏览器完全跳过其内部的布局和绘制计算，进一步降低后台开销。

## 综合/通用高频八股

### 1. TypeScript：`type` 和 `interface` 的区别？

**答案：**

- **`interface`：** 专门用于描述对象的形状，支持**声明合并**（同名 interface 自动合并）和 `extends` 继承，适合定义公共 API 和第三方库的类型扩展。
- **`type`：** 更通用，除了对象还能定义联合类型、交叉类型、元组、映射类型等。不支持声明合并，但功能更灵活。
- **项目经验：** 一般组件 props 和 API 响应数据用 `interface`（方便扩展），工具函数的参数和联合类型用 `type`。

### 2. TypeScript：泛型怎么用？

**答案：**

- 泛型是**类型的参数化**，让函数/类/接口可以处理多种类型而不丢失类型信息。
- 常见用法：`function identity<T>(arg: T): T { return arg }`，调用时 `identity<string>('hello')` 或让 TS 自动推导。
- 项目中如请求函数 `request<T>(url): Promise<T>`，泛型让返回值类型随调用时指定而变化，既复用了逻辑又保持了类型安全。

### 3. Electron：主进程和渲染进程的区别？它们怎么通信（IPC）？

**答案：**

- **主进程（Main）：** 每个应用只有一个，运行 Node.js 环境，负责创建窗口（`BrowserWindow`）、管理系统级操作（菜单、托盘、文件系统等）。
- **渲染进程（Renderer）：** 每个窗口一个，运行 Chromium 环境，负责页面渲染。默认不能直接访问 Node.js API（安全沙箱）。
- **IPC 通信：** 通过 `ipcMain`（主进程监听）和 `ipcRenderer`（渲染进程发送）进行消息传递。现代做法是在 `preload.js` 中通过 `contextBridge.exposeInMainWorld` 安全地暴露 API 给渲染进程。
- **为什么桌面端更在乎 CPU：** 不像浏览器标签页被系统自动限制资源，Electron 应用是独立进程，后台高 CPU 占用会直接导致用户风扇狂转、电池耗尽，用户感知很强。

### 4. HTTP：歌曲封面的跨域问题怎么解决的？Canvas 里 `drawImage` 跨域会怎样？

**答案：**

- **Canvas 跨域污染：** 如果 `drawImage` 绑定了一张跨域图片且没有设置 CORS，Canvas 会变成"被污染"状态，调用 `toDataURL`/`toBlob`/`getImageData` 会直接报安全错误。
- **解决方式：** 给 `<img>` 标签设置 `crossOrigin = "anonymous"`，同时服务端响应头需包含 `Access-Control-Allow-Origin`。Electron 环境中由于主进程可以代理请求或直接操控网络策略，跨域问题相对容易处理。

### 5. Vite：热更新（HMR）原理？和 Webpack 的区别？

**答案：**

- **Vite HMR 原理：** 基于原生 ES Module。开发时不打包，浏览器直接请求源文件，Vite 的 dev server 按需编译。文件修改时，通过 WebSocket 通知浏览器只重新请求变化的模块，实现毫秒级热更新。
- **Webpack HMR：** 需要先把所有模块打包成 bundle，修改后重新编译整个依赖链（虽然有增量编译优化但还是慢），通过 `HotModuleReplacementPlugin` 替换模块。
- **核心区别：** Vite 冷启动快（不打包），HMR 快（只更新单个模块）；Webpack 冷启动慢（要打包），但生态成熟、配置灵活。Vite 生产环境用 Rollup 打包，Webpack 用自己的打包器。
